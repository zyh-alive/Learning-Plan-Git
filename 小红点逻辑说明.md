# 小红点逻辑 - 代码位置与调试说明

## 一、代码在哪

### 1. 前端（显示/隐藏小红点）

| 文件 | 行号 | 作用 |
|------|------|------|
| **public/dashboard.html** | **85-94** | CSS：`.red-dot` 红点样式，`.red-dot.hide` 隐藏 |
| **public/dashboard.html** | **109-110** | HTML：红点 DOM，默认无 `hide` 类（默认显示） |
| **public/dashboard.html** | **147-170** | JS：请求 `/auth/profile` → `/auth/users/:id`，用返回的 `u.isCompleted` 控制红点 |

**关键逻辑（dashboard.html 第 161-164 行）：**
```javascript
var isCompleted = u && u.isCompleted === true;
var dot = document.getElementById('redDot');
if (isCompleted) dot.classList.add('hide');   // 已完善 → 隐藏红点
else dot.classList.remove('hide');            // 未完善 → 显示红点
```

- 红点**显示**：没有 `hide` 类。
- 红点**不显示**：有 `hide` 类。
- 只有接口返回的 `user.isCompleted === true` 时才会加 `hide`。

---

### 2. 后端（计算 isCompleted）

| 文件 | 行号 | 作用 |
|------|------|------|
| **controllers/authController.js** | **17-25** | `computeIsCompleted(profile)`：客户五项全填才返回 true |
| **controllers/authController.js** | **187-192** | `getUser` 里根据 role 决定用 `computeIsCompleted(profile)` 还是 DB 的 isCompleted |
| **controllers/authController.js** | **194-199** | 把算好的 `isCompleted` 放进 `userData`，返回给前端 |

**客户“五项全填”判定（authController.js 第 18-24 行）：**
```javascript
const hasName = profile.name != null && String(profile.name).trim() !== '';
const hasBirth = profile.birth != null && String(profile.birth).trim() !== '';
const hasGender = profile.gender != null && profile.gender !== '';  // 0 也算已选
const hasBio = profile.bio != null && String(profile.bio).trim() !== '';
const hasAbout = profile.about != null && String(profile.about).trim() !== '';
return hasName && hasBirth && hasGender && hasBio && hasAbout;
```

**getUser 里 isCompleted 的取值（authController.js 第 187-192 行）：**
- 客户（user）：**只用** `computeIsCompleted(profile)`，不看数据库里的 isCompleted。
- 顾问（consultant）：用数据库的 `profile.isCompleted`。

---

## 二、数据流（谁调谁）

1. 打开 **dashboard.html** → 执行里面的 async 函数。
2. `GET /auth/profile`（带 token）→ 返回 `{ user: { userId, id, ... } }`。
3. 用 `profileData.user.userId` 调 `GET /auth/users/:userId`。
4. **authController.getUser** 里：
   - 用 token 的 role 取对应的 Profile 表；
   - 查当前用户的 profile；
   - 客户：`isCompleted = computeIsCompleted(profile)`；
   - 把 `isCompleted` 放进 `userData`，返回 `{ user: userData }`。
5. 前端拿到 `userData.user.isCompleted`，据此给红点加/去 `hide` 类。

## 三、自己调试时建议看什么

1. **前端**  
   - 在 dashboard 的 async 里加 `console.log('profileRes', profileRes.ok, profileData)` 和 `console.log('userRes', userRes.ok, userData)`，确认两次请求都成功且 `userData.user.isCompleted` 的值。
   - 再打 `console.log('isCompleted', isCompleted, 'dot.className', dot.className)`，确认是否按预期加/去 `hide`。

2. **后端**  
   - 在 **authController.js** 的 `getUser` 里、算完 `isCompleted` 后加一行日志（见下），看 profile 各字段和最终的 `isCompleted`。
   - 若 `isCompleted` 为 true 但实际没填全，就重点看 `computeIsCompleted` 里用到的 `profile.name / birth / gender / bio / about` 在数据库/模型里到底是啥类型、啥值（是否被存成空字符串、null、0 等）。

3. **接口直接测**  
   - 用浏览器开发者工具 Network 看：  
     - `GET /auth/profile` 的响应里的 `user.userId`；  
     - `GET /auth/users/<该 userId>` 的响应里的 `user.isCompleted`。  
   - 确认客户身份下，未填全五项时接口是否返回 `isCompleted: false`。

## 四、可能出问题的地方

- **profileData.user.userId 为 undefined**：前端会请求 `/auth/users/undefined`，接口 404，async 里 `!userRes.ok` 直接 return，不会执行红点逻辑，红点保持默认（当前默认是显示）。若你看到红点“一直不显示”，多半是后面把红点设成隐藏了，说明某处还是拿到了 `isCompleted: true`。
- **后端返回的 isCompleted 不对**：客户必须走 `computeIsCompleted(profile)`，且 profile 必须是当前用户的 UserProfile 那条记录；若拿错表或错用户，会算错。
- **Sequelize 字段名**：UserProfile 是 camelCase（name, birth, gender, bio, about），`underscored: true` 只影响表字段名，实例上仍是 `profile.name` 等，一般没问题。若怀疑拿不到值，可在 `computeIsCompleted` 里打 `console.log(profile && profile.toJSON ? profile.toJSON() : profile)` 看实际结构。
