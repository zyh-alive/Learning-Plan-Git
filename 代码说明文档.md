# 顾问端项目 — 代码说明文档（API 与控制器对照）

本文档按 **URL 前缀** 列出每个接口对应的 **控制器方法** 及 **业务功能**。  
更细的启动流程、WebSocket、Mermaid 总览见同目录 **`代码流程图.md`**。

**约定**

- 除特别说明外，「需登录」的接口在请求头携带：`Authorization: Bearer <JWT>`（由 **`middleware/auth.js`** → **`utils/verifyAccessToken.js`** 校验）。
- 根路径前缀：`/auth`、`/api/order`、`/api/reviews`、`/api/consultants`、`/api/recharge`、`/api/consultant/service` 均在 **`server.js`** 中挂载。

## 一、`/auth/*` — `routes/authRoutes.js` → `controllers/authController.js`

| 方法 | 路径 | 控制器方法 | 是否需要登录 | 功能说明 |
|------|------|------------|--------------|----------|
| POST | `/auth/check-phone` | `checkPhone` | 否 | 检查手机号是否已注册（用于注册流程分支）。 |
| POST | `/auth/send-code` | `sendCode` | 否 | 向未注册手机号发送验证码（开发环境多为控制台输出）。 |
| POST | `/auth/verify-code` | `verifyCode` | 否 | 校验验证码；通过仅提示后续可设密码，**不签发 token**。 |
| POST | `/auth/set-password` | `setPassword` | 否 | 设置密码并完成注册；**返回 JWT**，前端可进入主页。 |
| POST | `/auth/login` | `login` | 否 | 已注册用户登录；**返回 JWT**；payload 含 `role`（`user` / `consultant`）及 `token_version` 等。 |
| GET | `/auth/profile` | （内联在路由文件） | 是 | 返回当前 token 解析后的用户信息摘要（非 `authController` 内单独函数）。 |
| GET | `/auth/users/:id` | `getUser` | 是 | 查询指定用户/顾问资料（含金币等展示字段，受角色与 id 校验约束）。 |
| PUT | `/auth/users/:id` | `updateProfile` | 是 | 更新当前角色对应资料表中的昵称、简介等可写字段。 |
| POST | `/auth/change-password` | `changePassword` | 是 | 修改密码；同步递增资料表 **`token_version`**，使旧 JWT 失效。 |

**说明**：注册/登录通过 **`config/roleConfig.js`** 区分客户（`user_auth` / `user_profile`）与顾问（`consultant_auth` / `consultant_profile`），同一套路由、不同表。

## 二、`/api/order/*` — 订单聊天 HTTP（优先挂载）

**文件**：`ws/order-chat/orderChatHttpRoutes.js` → **`ws/order-chat/orderChatHttpController.js`**  
**挂载顺序**：在 **`server.js`** 中 **先于** `routes/orderRoute.js` 挂到 `/api/order`，避免被 `/:orderId/pay` 等动态路由抢先匹配。

| 方法 | 路径 | 控制器方法 | 是否需要登录 | 功能说明 |
|------|------|------------|--------------|----------|
| GET | `/api/order/chat/conversations` | `getChatConversations` | 是 | 当前用户/顾问的**会话列表**（含订单维度、最后一条消息、未读数等，规则见控制器内注释）。 |
| GET | `/api/order/:orderId/chat/messages` | `getOrderChatMessages` | 是 | 指定订单的**聊天记录分页列表**（需为订单参与方）。 |
| POST | `/api/order/:orderId/chat/messages` | `postOrderChatMessage` | 是 | **发送**一条聊天消息（通常要求订单处于可聊状态，如服务中等，以控制器校验为准）。 |
| POST | `/api/order/:orderId/chat/read` | `markOrderChatRead` | 是 | 将对方发来消息标为已读；可能触发 **`broadcastOrderChat`** 推送给在线端。 |

**实时通道**：同端口 **`/ws/order-chat`** 由 **`ws/order-chat/orderChatWsRoutes.js`**、**`orderChatWsController.js`** 处理，与上表 REST 共用消息表与广播逻辑（**`ws/order-chat/broadcast.js`**）。

## 十一（补充）、订单加急（pending_rush）

- 状态 **`pending_rush`**：客户已付订单款后又支付加急费；加急费 = 订单价一半（服务端计算）；默认加急接单窗口 **1 小时**（`rush_duration_seconds`，可传 60～86400）。
- **冻结**：本金 `freeze_records.purpose=main`（原因「用户付款」）；加急 `purpose=rush`（原因 **「加急付款」**）。
- **加急超时无人接单**：只退加急冻结，原因 **「加急过期」**；订单回到 **`pending`**，并将 **`expires_at` 设为 `created_at + 24h`**（普通单截止）。
- **接单 / 完成**：顾问接单后流程不变；完成结算时 **合并所有仍冻结中的款项**（本金+加急）一次性释放给顾问。
- 接口：`GET /api/order/:orderId/rush/preview`、`POST /api/order/:orderId/rush/pay`（须在 `POST .../pay` 路由之前注册）。
- 手工 SQL：见 **`sql/orders_rush_freeze_purpose.sql`**。

## 十一、订单接单存活时间与「过期」

- 字段（`orders` 表）：**`survival_seconds`**（默认 86400 = 24 小时）、**`expires_at`**（客户支付成功时写入 = 支付时刻 + 存活秒数）；超过 **`expires_at`** 仍未接单则 **`status = expired`**，并写 **`expired_at`**。
- 退款：`freeze_records` 追加 **`refunded`**，**`reason = '订单过期'`**（与顾客主动取消等区分）。
- 定时任务：`server.js` 启动后每分钟执行 **`orderController.runPendingOrderExpirySweep()`**；接单、订单详情、支付前也会尝试对单笔订单过期校验。
- 手工迁移（未用 `sync({ alter: true })` 时）：见 **`sql/orders_survival_expired.sql`**。
